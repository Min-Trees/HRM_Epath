# T03 — Lớp chung backend (lỗi, phân trang, audit, bảo mật cơ bản)

## Đã làm gì

Xây các thành phần nền dùng chung cho toàn bộ module backend, gói trong package `com.company.hrm.common`:

- **Lỗi thống nhất**:
  - `BusinessException` (409 Conflict mặc định, có thể override status) — dùng cho vi phạm quy tắc nghiệp vụ (vd `BANG_CONG_DA_CHOT`, `PAYROLL_LOCKED`).
  - `ResourceNotFoundException` (404) — khi truy vấn không thấy.
  - `ForbiddenException` (403) — khi thiếu quyền.
  - `ErrorResponse` — payload lỗi thống nhất `{ timestamp, code, message, details[] }` với `details` chứa danh sách `FieldError(field, message)` cho validation.
  - `GlobalExceptionHandler` (`@RestControllerAdvice`) map các exception → `ErrorResponse`. Đã test.

- **Phân trang**:
  - `PageResponse<T>` — DTO chuẩn `{ content, page, size, totalElements, totalPages }`, có `from(Page<S>, Function<S,T>)`.

- **Audit**:
  - `BaseAuditEntity` — `@MappedSuperclass` với `createdAt`, `updatedAt` (Hibernate `@CreationTimestamp` / `@UpdateTimestamp`) + `createdBy`, `updatedBy` (UUID, fill qua listener).
  - `AuditingEntityListener` — `@PrePersist` / `@PreUpdate` đọc `AuthContext.currentUserIdOrNull()` để gắn `createdBy`/`updatedBy`. Dùng reflection an toàn (không lỗi nếu entity không có field này).

- **Tiền tệ**:
  - `MoneyUtils` — `BigDecimal` + `RoundingMode.HALF_UP`, scale 2. Có `money()`, `add/subtract/multiply`, `percent(amount, percent)` cho tính tỷ lệ BHXH/thuế, `isPositive()`. Cấm dùng `double` trong tính lương.

- **Bảo mật stub**:
  - `Role` enum: `HR, MANAGER, BHXH_OFFICER, PAYROLL_ACCOUNTANT, EMPLOYEE`.
  - `AuthContext` — `ThreadLocal` lưu `(userId, role)`, có `requireUserId()`, `requireRole(Role...)` throw `ForbiddenException`.
  - `@RequiresRole(Role...)` — annotation bảo vệ method/class.
  - `AuthAspect` — `@Around` mọi controller, đọc header `X-User-Id` (UUID) + `X-User-Role` (enum) → set vào `AuthContext` → check `@RequiresRole`. **Đây là stub dev** — sẽ thay bằng Spring Security + JWT khi tích hợp thật.

- **Tiện ích thời gian**:
  - `DateUtils.workingDaysBetween(from, to)` — đếm ngày làm việc (T2..T6), dùng cho quy tắc nghỉ không lương ≥ 14 ngày làm việc (BHXH).
  - `isWorkingDay`, `daysBetween`.

- **Cấu hình chung**:
  - `WebConfig` — CORS cho frontend dev (`http://localhost:5173` + `http://localhost:4173`), Jackson customizer (JavaTimeModule, ISO-8601, `FAIL_ON_UNKNOWN_PROPERTIES=false`, NON_NULL), `ObjectMapper` bean chuẩn.

- **Test** (4 file, 23 test):
  - `GlobalExceptionHandlerTest` (7) — mapping status/code cho Business/NotFound/Forbidden/Illegal/NotReadable/ErrorResponse.
  - `MoneyUtilsTest` (6) — rounding, percent, isPositive.
  - `DateUtilsTest` (5) — workingDays tuần đầy đủ, ngày lẻ, đảo chiều.
  - `AuthContextTest` (5) — requireRole thành công/thất bại, requireUserId.

## Quyết định thiết kế

1. **Stub auth thay vì Spring Security**: conventions yêu cầu "bảo mật cơ bản mức prototype" — chọn stub với header `X-User-Id` / `X-User-Role` để task sau có thể chạy và test ngay. Khi tích hợp thật sẽ thay bằng filter chain Spring Security + JWT, các task nghiệp vụ không cần đổi code chỉ cần đổi cơ chế set `AuthContext`.
2. **Mọi ngoại lệ có `code` dạng chuỗi** (`BANG_CONG_DA_CHOT`, `EMPLOYEE_NOT_FOUND`, ...) — để frontend i18n / hiển thị message chính xác mà không cần parse chuỗi tiếng Việt.
3. **`BusinessException` mặc định 409**: tuân đúng conventions — "409 vi phạm quy tắc nghiệp vụ". Vẫn cho phép truyền status khác (vd 422) khi cần.
4. **`@RequiresRole` check trong Aspect chứ không Spring Security**: vì đây là stub, đặt logic đơn giản trong Aspect, dễ thay thế khi có JWT. Method-level ưu tiên hơn class-level nếu cả hai có.
5. **`PageResponse.from(Page<S>, Function)`** thay vì trả `Page<T>` trực tiếp: tách representation khỏi Spring Data, dễ kiểm soát JSON shape.
6. **`AuditingEntityListener` dùng reflection**: tránh ép mọi entity phải có cả 2 field; nếu entity không có `setCreatedBy` thì bỏ qua. Phù hợp với schema hiện tại (một số bảng chỉ có `created_by` không có `updated_by`).
7. **DateUtils đếm "ngày làm việc" = T2..T6** (chưa tính lễ/Tết): đơn giản cho T03, các task sau (T13 BHXH) nếu cần tính chính xác sẽ bổ sung bảng `danh_sach_ngay_le`.

## Quy tắc nghiệp vụ đã áp dụng

- **Mọi tiền tệ qua `MoneyUtils`** → scale 2, HALF_UP. Chuẩn bị cho T14, T15.
- **`GlobalExceptionHandler` ưu tiên 409 cho BusinessException**: áp dụng cho T07 (bảng công chốt), T11 (chốt/hủy chốt), T13 (báo BHXH không hợp lệ), T15 (chạy lương khi bảng công chưa chốt).
- **Audit auto-fill `createdBy`/`updatedBy`** từ `AuthContext`: chuẩn bị sẵn, dùng cho mọi entity nghiệp vụ từ T04 trở đi.
- **Headers `X-User-Id` / `X-User-Role`** đã chuẩn — frontend sẽ gửi kèm mọi request (T17).

## Ghi chú cho task sau

- **T04 trở đi**: mọi entity nghiệp vụ nên kế thừa `BaseAuditEntity` nếu bảng có `created_at/updated_at`. Lưu ý `BienDongNhanSu`, `BhxhBienDong` (append-only) **không có trigger auto-update ở DB** (xem V1) — nếu cập nhật sẽ vi phạm append-only, vì vậy KHÔNG nên dùng `@UpdateTimestamp` trên các entity này (có thể tạo subclass `BaseAppendOnlyEntity` không có `updatedAt`, hoặc cố ý ignore field).
- **Stub auth**: Khi tích hợp JWT thật, thay thế `AuthAspect` bằng filter `OncePerRequestFilter` đọc JWT → set `AuthContext`. API controller không cần đổi. `@RequiresRole` chuyển thành `@PreAuthorize` trong Spring Security.
- **`AuthAspect` dùng `MethodSignature` từ `org.aspectj.lang.reflect`**: cần import đầy đủ. Đã sửa trong task này.
- **`MoneyUtils.percent` ban đầu quên normalize scale**: đã sửa, giờ luôn trả về scale 2. Task T14/T15 sẽ test thêm với phép tính phức tạp (OT 150/200/300%, BHXH 10.5/21.5%).
- **DateUtils chưa tính ngày lễ/Tết**: nếu T13 cần tính chính xác nghỉ không lương ≥ 14 ngày làm việc (có trừ lễ), bổ sung bảng `danh_sach_ngay_le` + tham số vào `DateUtils`.
